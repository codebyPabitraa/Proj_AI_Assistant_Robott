# AI Assistant Robot — System Design Overview

## Project Summary

This is a comprehensive design documentation for an autonomous AI assistant robot deployed in an office environment. The robot is capable of voice interaction, autonomous navigation, meeting transcription, and real-time emotional feedback through animated eyes.

**Target Hardware:** Raspberry Pi 4 (8GB) + custom chassis
**Deployment:** On-premises office environment
**Timeline:** 6-month MVP development

## Key Design Decisions

### Why Raspberry Pi 4?
- Good CPU/cost ratio for hobbyist + SME projects
- Native GPIO support for motor control
- Proven ROS2 support
- Can run lightweight LLMs (Ollama + LLaMA 3.2 3B)
- Abundant documentation & community

### Why Ollama + LLaMA 3.2 3B?
- Runs offline (no cloud dependency or latency)
- Lightweight enough for Pi (3B Q4 = 2.5GB VRAM)
- Good reasoning for task planning
- Free and open source
- ~3-8s response time acceptable for robot

### Why Custom Pathfinding (A*) vs. ROS2 Navigation Stack?
- Simpler to debug and tune for office environment
- Lower CPU overhead on Pi
- Faster iteration during development
- Can upgrade to ROS2 nav2 in production

### Why Inductive Charging?
- No exposed contacts (safe)
- No manual alignment needed (after docking)
- Weatherproof
- Professional appearance

### Why MQTT + WebSocket?
- MQTT: Low-bandwidth, reliable for sensor data
- WebSocket: Real-time UI updates
- Both: Event-driven architecture, loose coupling

## Performance Targets

| Metric | Target | Notes |
|--------|--------|-------|
| Wake word latency | < 100ms | On-device, always listening |
| STT latency | 2-5s | 5s audio chunk |
| LLM response time | 3-8s | LLaMA 3B Q4 on Pi |
| TTS latency | 1-3s | Piper pre-synthesized |
| Total interaction | 8-15s | Wake → response → speech |
| Navigation update rate | 10 Hz | Path planning + motor control |
| Battery runtime | 6-8 hours | At 50% motor duty cycle |
| WebSocket latency | < 50ms | Telemetry streaming |

## Known Limitations (MVP)

1. **LLaMA inference slow** (3-8s) → Upgrade to Jetson Orin in production
2. **Single-threaded audio** → Use separate thread for LiDAR during TTS
3. **No multi-robot support** → Would need Redis + centralized state
4. **Limited vision in low light** → Add IR sensors or lighting
5. **Manual face enrollment** → No automatic face learning yet
6. **Meeting summaries generic** → Need domain-specific fine-tuning

## Troubleshooting Guide

### Robot won't move
- Check motor power (12V rail)
- Verify encoder connections
- Test motor directly via GPIO

### Voice recognition failing
- Check microphone levels (arecord -l)
- Verify Whisper model downloaded
- Test wake word in quiet environment

### Docking not aligning
- Verify IR beacon brightness
- Check camera image quality
- Reduce alignment speed

### High CPU usage
- Check for runaway threads (ps aux)
- Monitor Ollama memory (ollama ps)
- Reduce LiDAR scan rate
